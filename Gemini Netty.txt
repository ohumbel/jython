Diese spezifische Fehlermeldung in Netty ist oft etwas irreführend, da sie technisch gesehen nur besagt, dass der SslHandler aus der Pipeline entfernt wurde, während er eigentlich noch damit beschäftigt war, die verschlüsselte Verbindung aufzubauen.

Das ist meistens kein Fehler im SSL-Code selbst, sondern ein Symptom dafür, dass die Verbindung vorzeitig geschlossen wurde. Hier sind die häufigsten Ursachen und wie du sie behebst:

1. Vorzeitiges Schließen der Verbindung (TCP Close)
Die häufigste Ursache ist, dass die Gegenseite (Client oder Server) die TCP-Verbindung hart abbricht, während der TLS-Handshake noch läuft.

Ursache: Ein Firewall-Timeout, ein Load Balancer oder die Gegenseite stellt fest, dass sie dem Zertifikat nicht vertraut und bricht sofort ab.

Lösung: Prüfe die Logs der Gegenseite. Wenn es ein Client ist, schau nach "Connection reset by peer" oder Zertifikatsfehlern.

2. Fehlkonfiguration in der Netty Pipeline
Wenn du die Pipeline dynamisch änderst, kann es sein, dass dein Code den SslHandler versehentlich entfernt oder den Channel schließt, bevor der Handshake-Future abgeschlossen ist.

Prüfung: Suchst du in deinem Code nach pipeline.remove(...) oder ctx.close()?

Lösung: Warte immer auf das Resultat des Handshakes, bevor du die Pipeline umbaust:

Java
sslHandler.handshakeFuture().addListener(future -> {
    if (future.isSuccess()) {
        // Jetzt ist es sicher!
    } else {
        // Handshake fehlgeschlagen
    }
});
3. Protokoll-Mismatch (HTTP vs. HTTPS)
Ein klassischer Fehler: Ein Client versucht via HTTP (unverschlüsselt) auf einen Port zuzugreifen, der HTTPS (Netty mit SslHandler) erwartet.

Was passiert: Der SslHandler empfängt Daten, die nicht nach TLS aussehen, erkennt den Fehler und Netty schließt den Channel bzw. entfernt den Handler.

Lösung: Sicherstellen, dass der Client wirklich https:// nutzt.

4. Idle State Timeouts
Hast du einen IdleStateHandler in der Pipeline?

Ursache: Wenn der Handshake (warum auch immer) länger dauert als dein definiertes Timeout (z. B. wegen hoher CPU-Last oder langsamer Entropie-Generierung für Keys), löst der IdleStateHandler ein Schließen des Channels aus.

Lösung: Timeout-Werte während der Entwicklungsphase testweise erhöhen.

Zusammenfassung der Diagnose-Schritte
Zertifikate prüfen: Nutzt du ein selbstsigniertes Zertifikat? Falls ja, muss die Gegenseite dieses explizit akzeptieren.

Netty Logging: Aktiviere das Debug-Logging für SSL, um zu sehen, was genau während des Handshakes passiert:

Setze das System-Property: -Djavax.net.debug=ssl,handshake

Pipeline-Reihenfolge: Der SslHandler muss in der Regel als allererstes (bzw. direkt nach einem optionalen Proxy-Handler) in der Pipeline stehen.

Kurztipp: Schau dir in deinem Stacktrace besonders Zeile 729 im SslHandler an. Dort wird handlerRemoved0 aufgerufen, weil der Channel inaktiv wurde. Das Problem liegt also fast immer "außerhalb" des Handlers – die Verbindung unter ihm ist einfach weggebrochen.

Das Spannende an diesem Log ist tatsächlich nicht das, was drinsteht, sondern wo es aufhört.

Der Log zeigt die komplette Initialisierungsphase des SSL-Kontexts (Laden der Cipher Suites, Einlesen des Truststores mit 144 Zertifikaten, Seeding von SecureRandom), aber er bricht ab, bevor das erste Byte des eigentlichen Handshakes (das ClientHello) gesendet oder empfangen wurde.

Hier ist die Analyse der "verdächtigen" Punkte:

1. Das abrupte Ende (Der "Silent Death")
Normalerweise müssten nach dem Seeding von SecureRandom Einträge folgen wie:
[PRODUCING] ClientHello oder [RECEIVING] ServerHello.
Dass sofort die Exception kommt, bedeutet: Die Verbindung wurde geschlossen, noch bevor Netty überhaupt das erste TLS-Paket auf die Leitung schicken konnte.

2. Die Seeding-Wiederholungen
Plaintext
javax.net.ssl|ALL|...|trigger seeding of SecureRandom
javax.net.ssl|ALL|...|done seeding of SecureRandom
Dass dies viermal hintereinander passiert, ist bei Netty oft ein Zeichen dafür, dass mehrere Channels gleichzeitig versucht haben zu starten oder die Pipeline mehrfach initialisiert wurde. Das ist an sich kein Fehler, deutet aber auf eine hohe Dynamik beim Verbindungsaufbau hin.

3. Wahrscheinlichste Ursache: "Racing Condition" in der Pipeline
Da die Fehlermeldung SslHandler removed before handshake completed lautet und im Log keinerlei TLS-Protokoll-Aktivität zu sehen ist, passiert folgendes:

Netty öffnet den TCP-Channel.

Der SslHandler wird hinzugefügt.

Der SslContext wird initialisiert (das ist das, was du im Log siehst).

Noch während die Initialisierung läuft oder unmittelbar danach, wird der Channel von einer anderen Komponente (oder dem Betriebssystem/Remote Peer) geschlossen.

Netty sieht: "Channel inaktiv" -> "Entferne alle Handler" -> SslHandler wirft die Exception, weil er noch nicht fertig war.

Was du jetzt prüfen solltest:
Netzwerk-Ebene: Versuche einen simplen Verbindungstest mit curl -v https://dein-ziel oder openssl s_client -connect dein-ziel:port. Wenn das auch sofort scheitert, liegt es an der Firewall oder dem Routing, nicht an Java.

Port-Verwechslung: Bist du sicher, dass die Gegenseite auf diesem Port TLS spricht? Wenn du versuchst, TLS gegen einen Plain-HTTP-Port zu sprechen, schließen manche Server die Verbindung sofort, wenn sie die ersten "komischen" (binären) Bytes sehen.

Netty Version & EventLoop: Nutzt du eine sehr alte Netty-Version oder mischst du verschiedene Versionen (z.B. durch transitive Abhängigkeiten)? Das handlerRemoved0 in deinem ersten Stacktrace deutet darauf hin, dass Netty den Handler aktiv wegen eines Channel-Events rauswirft.

Zusammenfassend: Dein Java/Zertifikats-Setup sieht bis zum Abbruch sauber aus. Das Problem ist mit 90%iger Wahrscheinlichkeit ein Netzwerk-Event (TCP Close), das von außen kommt oder durch einen fehlerhaften Zustandsautomaten in deiner Netty-Pipeline ausgelöst wird, noch bevor das erste verschlüsselte Paket gesendet wurde.

Warum bricht Java/Netty so früh ab?
Während curl erst nach dem Empfang des Zertifikats "meckert", wirft Netty die Exception oft schon früher. Wenn dein Java-Code so konfiguriert ist, dass er den Handshake sofort abbricht, wenn die Zertifikatskette nicht validiert werden kann, sieht das in der Pipeline genau so aus, wie du es erlebst: Der Handler wird entfernt, weil die TLS-Engine den Fehler intern bereits beim ersten Validierungsversuch wirft.

    .build();

Das openssl-Log ist die "Smoking Gun" – also der eindeutige Beweis. Es bestätigt schwarz auf weiß, was passiert:

Verbindung steht: Auf Port 443 meldet openssl sofort CONNECTED. Der Server ist also da und erreichbar.

Zertifikats-Fehler: OpenSSL meldet Verification error: self-signed certificate und Verify return code: 18.

TLS-Handshake klappt trotzdem (technisch): Da OpenSSL ein Diagnose-Tool ist, bricht es nicht sofort ab, sondern zeigt dir die Details an (RSA 4096 Bit, TLS v1.3).

Was bedeutet das für dein Netty-Problem?
In deinem Java-Programm ist der SslHandler strenger eingestellt als das openssl-Kommandozeilentool. Sobald der Server sein Zertifikat schickt, prüft die Java-Laufzeitumgebung dieses gegen den lokalen Truststore (cacerts). Da es selbstsigniert ist, sagt Java: "Halt! Diesem Server vertraue ich nicht."

Netty reagiert darauf, indem es die Verbindung sofort kappt. Da dies mitten im Prozess geschieht, wird der SslHandler entfernt, bevor er "Erfolg" melden konnte – und das löst die eingangs erwähnte SSLHandshakeException aus.

import io.netty.channel.*;
import io.netty.channel.nio.NioEventLoopGroup;
import io.netty.channel.socket.SocketChannel;
import io.netty.channel.socket.nio.NioSocketChannel;
import io.netty.handler.ssl.SslContext;
import io.netty.handler.ssl.SslContextBuilder;
import io.netty.handler.ssl.util.InsecureTrustManagerFactory;

public class NettySslTest {
    public static void main(String[] args) throws Exception {
        String host = "self-signed.pythontest.net";
        int port = 443;

        // 1. SslContext erstellen (erlaubt TLS 1.2 und 1.3)
        final SslContext sslCtx = SslContextBuilder.forClient()
                .trustManager(InsecureTrustManagerFactory.INSTANCE)
                .build();

        EventLoopGroup group = new NioEventLoopGroup();
        try {
            Bootstrap b = new Bootstrap();
            b.group(group)
             .channel(NioSocketChannel.class)
             .handler(new ChannelInitializer<SocketChannel>() {
                 @Override
                 public void initChannel(SocketChannel ch) {
                     ChannelPipeline p = ch.pipeline();
                     // SslHandler muss ZUERST kommen
                     p.addLast(sslCtx.newHandler(ch.alloc(), host, port));
                     p.addLast(new SimpleChannelInboundHandler<Object>() {
                         @Override
                         public void channelActive(ChannelHandlerContext ctx) {
                             System.out.println("TCP Verbindung steht. Handshake läuft...");
                         }
                         @Override
                         protected void channelRead0(ChannelHandlerContext ctx, Object msg) {
                             System.out.println("Daten empfangen!");
                         }
                         @Override
                         public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
                             cause.printStackTrace();
                             ctx.close();
                         }
                     });
                 }
             });

            System.out.println("Verbinde mit " + host + "...");
            b.connect(host, port).sync().channel().closeFuture().sync();
        } finally {
            group.shutdownGracefully();
        }
    }

In welchem netty Artefakt befindet sie sich (ich verwende netty 4.2.12.Final)?

Bingo! Das ist das "Missing Link". Du hast definitiv ein Abhängigkeitsproblem.

Die Klasse io.netty.channel.unix.UnixChannel befindet sich im Artefakt netty-transport-native-unix-common.

Warum passiert das ausgerechnet beim SSL-Handshake?
Netty versucht heutzutage sehr intelligent zu sein: Wenn es auf einem Linux- oder macOS-System läuft, prüft es, ob es native Bibliotheken (wie OpenSSL über JNI) nutzen kann, statt der oft langsameren Standard-Java-SSL-Implementierung.

Sobald der SslHandler hinzugefügt wird, prüft er, ob er optimierte native Funktionen für "Unix-Sockets" nutzen kann. Dabei greift er auf die Klasse UnixChannel zu. Wenn diese im Klassenpfad fehlt, knallt es mit der NoClassDefFoundError.

Die Lösung: Benötigte Artefakte
Wenn du Netty manuell zusammenstellst (was bei Jython-Projekten oft der Fall ist), fehlen dir wahrscheinlich die "native" Brücken. Du solltest sicherstellen, dass folgende Artefakte im Klassenpfad liegen:

netty-transport-native-unix-common-4.1.112.Final.jar (Hinweis unten zur Version)

netty-common-4.1.112.Final.jar (Basis)

Wichtiger Hinweis zur Version: Du hast oben geschrieben, dass du 4.2.12.Final verwendest. Netty 4.2 ist zum jetzigen Zeitpunkt (2026) zwar im Umlauf, aber sehr viele Projekte nutzen noch die extrem stabile 4.1.x-Serie. Stelle unbedingt sicher, dass alle Netty-JARs die exakt gleiche Version haben. Ein Mix aus 4.1 und 4.2 führt fast immer zu solchen Fehlern.

Wie du es fixst (Maven/Gradle)
Falls du ein Build-Tool nutzt, füge das entsprechende native Paket hinzu. Für macOS/Linux ist das oft:

Maven:

XML
<dependency>
    <groupId>io.netty</groupId>
    <artifactId>netty-transport-native-unix-common</artifactId>
    <version>4.2.12.Final</version>
</dependency>
Für Jython/Manuell:
Lade dir die netty-transport-native-unix-common-4.2.12.Final.jar herunter (z.B. von Maven Central) und füge sie deinem Klassenpfad hinzu.

"Der schnelle Hack" (Falls du keine JARs hinzufügen kannst)

Hier ist die Aufschlüsselung für deine Plattform-Fragen:

1. Brauchst du unix-common für Ubuntu?
Ja, absolut.
Das Artefakt netty-transport-native-unix-common enthält, wie der Name sagt, den gemeinsamen Code für alle Unix-basierten Systeme. Da Linux (Ubuntu) und macOS (Darwin) beide zur Unix-Familie gehören, ist diese JAR die absolute Grundvoraussetzung für beide Systeme, wenn Netty native Optimierungen nutzen will.

2. Brauchst du für Windows eine Brücke?
Nein.
Windows nutzt eine völlig andere Architektur (I/O Completion Ports statt Epoll/KQueue). Netty sucht auf Windows gar nicht erst nach io.netty.channel.unix.*. Dort wird standardmäßig die Java-NIO-Implementierung verwendet, es sei denn, du fügst explizit das (selten genutzte) netty-transport-native-iocp hinzu. Dein aktueller Fehler würde auf Windows also vermutlich gar nicht erst auftreten.

3. Was du für Ubuntu und Mac wirklich brauchst (Die Checkliste)
Damit Netty auf Intel Mac und Ubuntu stabil läuft (und nicht über fehlende Klassen stolpert), solltest du folgende JARs im Klassenpfad haben:

netty-common: Die Basis.

netty-transport-native-unix-common: Die Brücke, die dir gerade fehlt.

netty-handler: Enthält den SslHandler.

netty-buffer, netty-transport, netty-codec: Die Kern-Komponenten.

Pro-Tipp für native Performance:
Wenn du willst, dass es auf Ubuntu richtig schnell läuft, könntest du noch netty-transport-native-epoll (mit dem Classifier linux-x86_64) hinzufügen. Aber für den Anfang reicht es völlig, die unix-common JAR hinzuzufügen, damit der ClassNotFound-Fehler verschwindet.

Zusammenfassung für dein Jython-Setup:

Ein wichtiger technischer Hinweis:
Wenn du die JAR hinzugefügt hast und der Fehler NoClassDefFound verschwindet, aber der SSL-Handshake immer noch hakt, könnte Netty versuchen, die native OpenSSL-Bibliothek zu laden (via netty-tcnative). Da du aber oben im Log SSLContextImpl.java (das ist Standard-Java) gesehen hast, wird das Hinzufügen der unix-common JAR den "Absturz" des Handlers verhindern, und Netty wird brav das Standard-Java-SSL nutzen.

Warum passiert das auf Windows?
Normalerweise ist UnixChannel, wie besprochen, ein Unix-Konzept. In der Netty-Version 4.2.x wurde jedoch intern umstrukturiert. Der SslHandler in dieser Version hat eine engere Kopplung an bestimmte Hilfsklassen. Wenn der SslHandler initialisiert wird, scannt er (über Reflection oder direkte Zugriffe) nach verfügbaren Optimierungen.

In deinem Fall scheint das "handgestrickte" Setup dazu zu führen, dass die JVM versucht, die Klasse SslHandler vollständig zu verifizieren, dabei über die Referenz auf UnixChannel stolpert und – da die JAR fehlt – die gesamte Initialisierung des Handlers abbricht.

Die endgültige Lösung für den Klassenpfad
In Netty 4.x (besonders 4.2) gibt es eine "Fat-Jar"-Option, die genau für solche Fälle (Jython, manuelle Setups) gedacht ist, damit man nicht 20 Einzel-Dateien jonglieren muss.

Statt vieler kleiner JARs empfehle ich dir dringend, die netty-all-4.2.12.Final.jar zu verwenden.

Warum? Diese JAR enthält alle Plattform-Brücken (Unix, MacOS, Windows-Spezialitäten) und alle Codecs (HTTP, SSL, etc.) in einer einzigen Datei.

Der Vorteil: Es spielt keine Rolle mehr, ob du auf einem Intel Mac, Ubuntu oder windows-latest bist – alle Klassen, die Netty jemals "quer-referenzieren" könnte, sind vorhanden.

Falls du bei Einzel-JARs bleiben musst:
Wenn netty-all zu groß ist, benötigst du für Netty 4.2 zwingend diese Kombination im Klassenpfad, damit der SslHandler auf allen OS-Plattformen überlebt:

netty-handler-4.2.12.Final.jar

netty-buffer-4.2.12.Final.jar

netty-common-4.2.12.Final.jar

netty-transport-4.2.12.Final.jar

netty-codec-4.2.12.Final.jar

netty-transport-native-unix-common-4.2.12.Final.jar (Ja, auch für Windows, da die Klasse dort referenziert wird!)

Warum hat sich das Verhalten geändert?
Früher (Netty < 4.1.x) waren diese Abhängigkeiten oft optional ("optional dependencies"). In neueren Versionen und unter Java 21 ist die Klassen-Verifizierung strenger geworden. Wenn der SslHandler geladen wird und eine Methode besitzt, die UnixChannel im Argument hat, will Java wissen, was das für eine Klasse ist – auch wenn die Methode auf Windows nie aufgerufen wird. Hier ist die logische Kette, was in deiner GitHub Pipeline passiert:

1. Das "Silent Failure" Muster
In Netty 4.2 führt ein interner Fehler (wie eine fehlende Klasse oder eine Exception bei der Zertifikatsprüfung) dazu, dass der SslHandler sich selbst aus der Pipeline entfernt, um den Channel nicht zu blockieren.

Der Effekt: Netty wirft die SSLHandshakeException, weil der Handler "weg" ist, bevor er success melden konnte.

Die Ursache: Meistens liegt ein Fehler in der initChannel-Methode vor, oder die SSLEngine bricht wegen der InsecureTrustManagerFactory Konfiguration ab.

2. TLS 1.3 und GitHub Actions (Temurin 21)
Da du Java 21 nutzt, erzwingt Netty standardmäßig modernste Sicherheitsfeatures. GitHub Pipelines sind oft sehr restriktiv.

Mögliches Problem: Wenn deine Jython-Module die InsecureTrustManagerFactory nutzen, aber gleichzeitig versuchen, Parameter zu setzen, die nur für TLS 1.2 gültig sind (z.B. bestimmte Cipher Suites), kollidiert das mit dem TLS 1.3 Standard von Java 21.

3. Diagnose-Checkliste für die Pipeline
Da du sagst, es ist "handgestricktes" Jython, prüfe folgende drei Punkte:

Punkt	Prüfung	Warum wichtig?
Pipeline Order	Ist der SslHandler wirklich der erste?	Wenn ein anderer Handler (z.B. ein Logger) vorher die Bytes "frisst", kann der SslHandler den Handshake nicht starten.
Zertifikats-Typ	Ist self-signed.pythontest.net die Ziel-URL?	Wie wir bei curl sahen, ist das Zertifikat dort "self-signed". Ohne funktionierenden TrustManager bricht Java 21 den Handshake sofort ab.
SniHelper	Nutzt du SNI?	Java 21 ist bei SNI (Server Name Indication) strenger. Wenn der Hostname im SslHandler nicht exakt mit der URL übereinstimmt, wird die Verbindung oft sofort getrennt.
